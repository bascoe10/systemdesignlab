# Contributing to SystemDesignLab

## The One Rule

**All work happens on `main`.** The `level-N-*/<system>` branches are
generated by CI on every push to main (`generator/generate.sh`) and
force-pushed — anything committed to them directly will be overwritten.

## Repo Model

```
systems/<name>/
  shared/            code + compose + load tests shared by all levels
    services/        one Go module; per-service main packages + internal/
    docker-compose.yml
    system.yaml.tmpl
  level-1-observe/   per-level overlay: docs + config.yaml
  level-2-experiment/
  level-3-build/     + stubs/ (replaces the real component on that branch)
  level-4-fix/       + .solutions/ (hidden until `make reveal-solution`)
  level-5-scratch/   + stubs/ (skeleton mains) + contracts/ (OpenAPI)
```

The generator assembles `infrastructure/ + cli/ + chaos-toolkit/ +
shared/ + level overlay` into a self-contained branch where the system
lives at `system/`. Run any level locally without touching branches:

```bash
make start SYSTEM=url-shortener LEVEL=3   # assembles into .lab/ and runs
```

## Development Loop

```bash
make test                                  # unit tests + vet + CLI build
make start LEVEL=1 && make load-test       # end-to-end via the generator
bash generator/generate.sh --out /tmp/lab url-shortener 5
(cd /tmp/lab/system/services && go build ./...)   # stub levels must compile
```

CI enforces: services tests (with `-race`), CLI build, shellcheck, compose
config validation for every level × both cache profiles, stub-level
compilation, and that `dashboards/*.json` match `gen_dashboards.py` output.

## Common Contributions

**Quiz questions** — add to `cli/diagnose/questions.yaml` with a
`maps_to_level` tag, and update the bucket counts in `scoring.yaml`.

**Grafana dashboards** — edit
`infrastructure/observability/grafana/gen_dashboards.py` and re-run it;
never edit the JSON by hand. Dashboards must stay identical across levels,
and any new panel needs the backing metric emitted from
`internal/obs/metrics.go` (metric names are a public API — the dashboards
and `cli/validate.go` depend on them).

**Chaos scripts** — pair every `chaos-toolkit/scripts/<x>.sh` with a
`resilience-challenges/<X>.md` containing observation prompts and success
criteria.

**New systems (Phase 2+)** — read [docs/SYSTEM_AUTHORING.md](docs/SYSTEM_AUTHORING.md)
first; it codifies the invariants (quantitative L3 tests, config-only
dashboard-diagnosable L4 misconfigs, baseline-relative thresholds, the
narrative thread) and the definition of done for a system PR. Note the
go/no-go gate in the README roadmap before investing.

## Style Notes

- Teaching docs state expected outcomes and ask questions; they don't just
  give answers. Hints escalate behind `<details>` folds.
- Level 4 root causes must be fixable in `config.yaml` — no code edits.
- Shell scripts: `bash`, `set -euo pipefail`, shellcheck-clean, LF endings
  (enforced by `.gitattributes`).
